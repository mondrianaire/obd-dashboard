# Project notes for Claude

This file orients any Claude agent that lands in this repo. It's written for the agent to read first, before touching code.

## What this is

An OBD-II telemetry dashboard for a 2016 VW Golf GTI (VIN `3VW5T7AU2GM058168`), deployed to GitHub Pages at https://mondrianaire.github.io/obd-dashboard/. It ingests CSV logs from OBD Fusion (live, syncs via Dropbox) plus a snapshot of Car Scanner `.dlg` SQLite files, derives per-drive and fleet-wide telemetry views, and surfaces engine-health and driver-style insights.

## Read patterns for large files

`index.html` has **~3,549 lines** (HTML structure + embedded CSS + the Dashboard class JavaScript). The Read tool's default limit is 2,000 lines, so a naive `Read(file_path="index.html")` will truncate.

When reading `index.html`, always pass an explicit `limit`:

```
Read(file_path=".../index.html", limit=4000)
```

Same applies to the bulky test file (`tests/test_dataviews.py`, 496 lines — under the default but worth the explicit limit for full reads) or any future module that grows past 2,000 lines.

The Read tool's `limit` parameter has no hard cap. Use it.

For surgical edits where you only need to see a specific section (chart definition, builder method, etc.), prefer `Grep` to locate the line number, then `Read` with `offset` and a small `limit` window.

## Files and what they do

```
index.html              Dashboard markup + CSS + the Dashboard class JS
                        (Loads telemetry_data.js for the dataset.)
telemetry_data.js       window.RAW = { drives: [...], totals: {...}, ... };
                        Generated by refresh_dashboard.py from telemetry_data.json.
telemetry_data.json     Canonical data file written by prep_drives.py.
                        Read by the test suite + injected into telemetry_data.js.
prep_drives.py          Scans Dropbox + data/ snapshot, builds telemetry_data.json.
                        Handles 3 source formats: OBD Fusion CSV (20-col + 100-col
                        v2 profile), Car Scanner .dlg SQLite, third-party CSV.
                        Vectorized pandas bucketing for performance.
archive_old_csvs.py     Pre-step rotation: materialize Dropbox online-only files +
                        archive files matching (rank > KEEP_NEWEST OR age >
                        ARCHIVE_AFTER_DAYS) to data/obd_fusion/. Defaults: keep
                        newest 1 file in Dropbox, anything older moves to local.
refresh_dashboard.py    One-shot orchestrator: rotate -> prep -> inject ->
                        verify -> regression-suite -> git commit/push.
                        Called by auto_refresh.ps1 from Windows Task Scheduler.
auto_refresh.ps1        Scheduled-task wrapper (runs at 7am + 7pm). Logs to
                        refresh.log.
setup_scheduled_refresh.ps1  One-time installer for the Windows scheduled task.
tests/test_dataviews.py Regression test suite: bounds, consistency, shifts,
                        aggregates, anomalies. Wired into refresh_dashboard.py
                        so every refresh validates. Exit 0 = all PASS.
data/                   Local archive of CSVs rotated out of Dropbox.
                        Read by prep_drives.py alongside the Dropbox folder.
                        gitignored.
```

## Data pipeline

```
phone (OBD Fusion app)
  -> Dropbox sync (~/Dropbox/Apps/OBD Fusion/CsvLogs/<VIN>/)
  -> archive_old_csvs.py (materialize cloud files, rotate old to data/)
  -> prep_drives.py (parse 3 source formats, dedupe, bucket to 1Hz, write JSON)
  -> refresh_dashboard.py (inject -> telemetry_data.js, bump badge)
  -> regression suite (validate)
  -> git push
  -> GitHub Pages serves index.html + telemetry_data.js
```

A normal refresh on the Windows machine takes ~15 seconds end-to-end.

```

               elevation profile, KPI cards

               boost vs RPM, torque/power curve, ignition vs boost,
```

The default sub-tab on first visit is **Drivetrain** in both views. On mobile, the default top-level view is **Fleet**.

## Conventions

- **Versioning**: dashboard version is in `refresh_dashboard.py` (`VERSION = "v2.x"`) and surfaced in the corner badge. Bump on any meaningful UI/feature change. Changelog lives in the comment block above `VERSION`.
- **Bumping conventions**: minor patches (bug fixes, tile reorderings) bump the minor digit; major reorganizations or new feature groups bump the first digit.
- **All file I/O is UTF-8** with `newline="\n"`. Windows Task Scheduler runs under cp1252 by default; the scripts force UTF-8 stdout via `sys.stdout.reconfigure`.
- **Sub-tab `data-subtab` keys** stay stable across renames (engine, diag, fuel, drivetrain, overview) so localStorage persistence survives label changes. Visible label changes are HTML-only.
- **Channel availability** is tracked in `d.channels.<key>` and gated in the HTML via `data-needs="<key>"`. The renderDrive() JS toggles `.disabled` on chart-containers when the drive lacks a needed channel.
- **Chart canvas IDs** for fleet aggregates are prefixed `fleet*` (e.g. `fleetRpmGearChart`). Per-drive canvases have no prefix.

## Adding a new tile (checklist)

1. **HTML**: add `<div class="chart-container">` (with `data-needs` if applicable) inside the right `<div class="subtab-panel">`. Both Fleet and Drive panels exist independently — make sure you target the right one.
2. **prep_drives.py**: if the tile needs a new channel, add it to the canonical column list, the OBD_FUSION_V2_COLS map (or DLG/third-party parsers), the bucket aggregator, and the row emission. Add a channel flag (`channels[<key>]`) and a summary peak if relevant.
3. **JS chart builder**: add a `this.charts.<name> = new Chart(...)` block inside a `buildVxxCharts()` method (group by version).
4. **JS update method**: feed data into the chart inside an `updateVxxCharts()` method. Call from `renderDrive()`.
5. **Tests**: add bound or consistency check to `tests/test_dataviews.py` if the channel has a known physical range.

## Test suite

```
python tests/test_dataviews.py
```

Reads `telemetry_data.json` (canonical, not the JS-wrapped version). Exit 0 = all PASS. Cron-friendly. The suite is also invoked by `refresh_dashboard.py` as step `[3.5/4]` and logs `SUMMARY: N PASS, N WARN, N FAIL` to `refresh.log`.

The suite intentionally re-implements gear inference, regime classification, and ideal-landing-RPM math in Python — that way it's a cross-check against the JS implementation, not a tautology.

## Known wrinkles

- The car has a known broken **G266 oil level/temperature sensor** (dash warning). The 100-col v2 OBD Fusion profile correctly shows no oil temp data; that's expected, not a logging bug.
- April 29 third-party drive has mis-decoded coolant/load/STFT — values shown as-logged with a caveat in the footer.
- Drives shorter than 10s are dropped during prep.
- Knock event definition: `boost > 2 psi AND ign < -5° AND throttle > 30%`. The throttle gate was added in v2.5 to exclude post-WOT lift-off artifacts where the ECU parks ignition during fuel cut.
- Shift regime classification uses physics-derived landing RPM (`fromRpm * (toRatio/fromRatio)`), not the noisy 1-second-later sample. This was a v1.9 fix.
